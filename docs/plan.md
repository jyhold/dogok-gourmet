>
> 작성일: 2026-07-12 · **v1.12 변경: 최상위 분기 점심/저녁 → 점심/후식 (저녁·번개모임·팀회식 폐지). 후식은 하위 분기 없이 단일 모드 + 브라우저 현재 위치 반경 300m(군인공제회관 폴백 시 500m, 부족 시 1km 확장). 후식 전용 `coffee` 시트(평점 대신 recommended 추천 T/F) + 후식 카테고리 5종·도트 아이콘 5종.** · v1.10: 미식가 방문 검증(visited)+평점(rating) + '⭐ 인증 우선' 체크박스 + 하단 마스코트 날씨 안내 · v1.9: 앱 명칭 '도곡한 미식가' + 위치인식 제거(군인공제회관 고정) + 거리 재설정(도보 1.3km/따릉이 2km/택시 5km, 직선거리) · v1.8: Claude 토큰 최소화(§9.1) · 상태: Phase 1(로컬 목업) 개발 완료

## 1. 프로젝트 개요

| 항목 | 내용 |
|---|---|
| **누구를 위해** | 매일 점심(+저녁)마다 뭐 먹을지 고민하는 사내 지인·직장 동료들 — 기준점: **군인공제회관 (서울 강남구 남부순환로 2806, 양재역 인근)** |
| **서비스 지역** | 기본 **서초구·강남구**. 거리 필터를 '택시'로 선택하면 **동작구·송파구까지 확장** |
| **왜 만드나** | 점심시간이 다가오거나 동료와 점심 약속이 있을 때, 어딜 갈지·뭘 먹을지 결정 못 하는 결정장애 현대인을 위한 앱 |
| **성공 기준** | 최소한의 제한 조건(필터)만 걸고 실행하면, 슬롯머신처럼 돌아가다 멈추며 맛집 1곳을 랜덤 추천. 웹앱 형태로 실제 배포까지 완료 |
| **핵심 가치** | ① 결정 스트레스 제거 ② 재미(슬롯 연출) ③ 관리자(주영) 큐레이션 신뢰도 |

### 3.1 핵심 사용자 플로우

```
앱 접속
  → 🍚 식사 모드 선택: [점심] / [후식]  ← 최상위 분기 (v1.12: 저녁 → 후식)
      ├ 기본 탭 자동 선택: 접속 시각 13시 전 → 점심, 13시 이후 → 후식 (탭으로 즉시 전환)
      ├ 점심 선택 시 하위 분기: [🍜 혼밥] / [👥 점심약속]
      │   ├ 혼밥: 1인 좌석·빠른 회전 업장 우대 — 혼밥 친화 카테고리(국밥·라멘·덮밥·분식 등) 가중치↑,
      │   │        관리자DB solo_friendly=TRUE는 추가 가중치, 고기구이·요리 중식당 등 다인 전제 카테고리 제외
      │   │  · 시작점 = 군인공제회관 고정 (위치인식 미사용)
      │   └ 점심약속: 기존 기본 동작 (동료·거래처와 함께 — 전 카테고리 후보)
      └ 후식 선택 시: 하위 분기 없이 바로 필터 (예외처리할 후식 메뉴 선택지만)
          · 현재 위치 반경 300m 카페·디저트 (군인공제회관 폴백 시 500m · 부족 시 1km 자동 확장)
          · 브라우저 Geolocation 사용 — 거부·실패 시 군인공제회관 폴백 + 안내 토스트
          · 카카오 CE7(카페) + coffee 시트. 예산·거리 필터 없음. '👍 미식가 추천 우선' 부스트
  → 날씨 자동 확인 (기상청 API)
      └ 하단 마스코트 말풍선에 현재 날씨 표시 + 악천후면 비 마스코트 (v1.14: 필터 자동 전환 없음 — 안내만)
  → 필터 세팅
      ├ (점심) ⭐ 미식가 인증 맛집 우선 / (후식) 👍 미식가 추천 우선 — 켜면 해당 후보 가중치 ×6 (확률 부스트)
      ├ 예산(점심만): 가성비(~1만) / 맛집(2~3만) / 플렉스(3만~) · 후식은 예산 필터 없음
      ├ 거리(점심만, 시작점=군인공제회관 직선거리): 도보(~1.3km) / 따릉이(~2km) / 택시(~5km) · 후식은 위치기반 자동(300m/폴백500m)
      └ 제외 메뉴: (점심) 2단계 카테고리 / (후식) 후식 5종 체크 (다중 선택)
  → 🎰 돌리기!
      └ 슬롯 릴 스핀 → 감속 → 1곳 멈춤
  → 결과 카드
      ├ 상호명 / 카테고리 / 거리·도보시간
      ├ 시그니처 메뉴 + 인당 가격대 (관리자DB 등록 맛집만)
      ├ 관리자 추천 맛집이면 👑 강조 배지 + 추천 코멘트
      ├ (점심) 미식가 평점 슬롯(별 5개, rating÷2) + '✅ 직접 방문 인증' 배지
      ├ (후식) '👍 미식가 추천' / '✅ 직접 방문 인증' 배지 (별점 없음)
      └ [네이버지도에서 보기] [다시 돌리기]
  → 다시 돌리기 (무제한, 단 이미 나온 곳은 후보군 소진 전까지 재등장 금지)
```

### 3.3 메뉴 카테고리 체계 (2단계)

관리자DB는 이 체계를 직접 지정하고, 카카오 검색 결과는 매핑 테이블로 변환한다.

| 대분류 | 하위 카테고리 |
|---|---|
| 한식 | 국밥·탕 / 고기구이 / 찌개·백반 / 칼국수 / 냉면·갈비탕 / 샤브샤브 / 죽·건강식 |
| 분식 | 떡볶이·김밥 / 국수·우동 / 만두 |
| 중식 | 짜장·짬뽕 / 마라·훠궈 / 양꼬치·중식당 |
| 일식 | 초밥·회 / 라멘·우동 / 돈카츠·카레 / 덮밥 / 장어요리 |
| 양식 | 파스타·피자 / 버거 / 스테이크·비스트로 / 멕시칸·타코 |
| 아시안 | 쌀국수·베트남 / 태국·팟타이 / 인도·커리 / 홍콩·딤섬 |
| 카페·간편 | 샐러드·포케 / 샌드위치·브런치 / 도시락 |
| 기타 | 뷔페·구내식당 / 매핑 실패 fallback |

> 하위 총 31개(v1.11 장어요리, 이후 아시안 홍콩·딤섬, 양식 멕시칸·타코 추가). 실사용 후 동료 피드백으로 증감 조정.

**후식(v1.12) — 별도 체계** (대분류 '후식' 고정, 카카오 CE7·`coffee` 시트 공용):

| 대분류 | 하위 카테고리 |
|---|---|
| 후식 | 커피·음료 / 베이커리·빵 / 케이크·디저트 / 도넛·와플 / 아이스크림·빙수 |

> 카카오 CE7(카페) category_name → 위 5종 매핑(`mapKakaoCafe`), 실패 시 '커피·음료'. 도트 아이콘 5종(coffee/bakery/cake/donut/icecream) 신규.

### 3.4 UI/디자인 컨셉 — Tiny Town 스타일 도트 디자인 (v1.6)

**컨셉 한 줄**: 강남 오피스 거리를 아기자기한 도트 마을로 — 점심시간마다 열리는 작은 픽셀 아케이드.

#### 비주얼 방향
- **스타일**: 16×16 도트 타일 감성 (Tiny Town 계열 에셋 팩에서 영감). 낮은 채도의 따뜻한 파스텔 팔레트 + 굵은 픽셀 아웃라인
- **마스코트**: 도트 캐릭터 1종 (예: 젓가락 든 꼬마 직장인) — 로딩·빈 결과·악천후 안내 등 상태 표현 담당
- **음식 도트 아이콘 세트**: 카테고리 24종별 16×16(표시 시 정수배 확대) 아이콘 — 국밥 뚝배기, 라멘, 초밥, 버거 등. 슬롯 릴에 이 아이콘들이 돌아감
- **슬롯머신**: 레트로 아케이드 기계를 도트로 그린 프레임 안에서 음식 아이콘+상호명이 스핀 → 당첨 시 도트 폭죽/반짝임 스프라이트 애니메이션
- **모드 선택 화면**: 4개 모드를 도트 일러스트 카드로 (혼밥=1인 테이블, 점심약속=마주 앉은 둘, 번개=번개 아이콘 든 캐릭터들, 팀회식=긴 테이블 잔치)
- **결과 카드**: 도트 프레임(9-patch 픽셀 보더) + 👑 관리자 추천은 도트 왕관 배지
- **악천후 연출**: 날씨 토스트에 도트 비구름·태양 스프라이트 — 기능 안내를 캐릭터 연출로

#### 타이포그래피
- **한글 픽셀 폰트**: Galmuri 또는 Neo둥근모(NeoDunggeunmo) — 무료 웹폰트, 도트 감성 유지하며 한글 가독성 확보
- **영문/숫자**: Press Start 2P (Google Fonts) — 가격·거리 등 숫자 강조용
- **가독성 규칙**: 헤드라인·버튼·라벨만 픽셀 폰트, 긴 본문(추천 코멘트 등)은 Pretendard 등 일반 폰트 혼용 — 작은 픽셀 폰트로 장문을 쓰면 모바일 가독성이 급락하므로

#### 기술 구현 규칙
- 도트 에셋은 원본 해상도로 저장 + CSS `image-rendering: pixelated`로 **정수배(2x/3x) 확대** — 블러 없는 또렷한 픽셀 유지
- 픽셀 보더·버튼은 9-patch 방식 또는 `box-shadow` 픽셀 테크닉으로 CSS 구현 (이미지 최소화)
- 애니메이션은 스프라이트 시트 + CSS `steps()` 타이밍 함수 — 도트 특유의 프레임 단위 움직임
- 다크모드는 MVP 제외 (팔레트 1벌로 시작)

#### 에셋 소싱 전략 (무료 경로 중심 — Higgsfield 미사용, 크레딧 소진 이슈)
1. **CC0 무료 에셋 활용 (메인)**: Kenney·OpenGameArt의 Tiny Town 계열 타일셋 — **라이선스(CC0/퍼블릭 도메인) 확인 후** 배경 타일·소품 등 범용 요소에 적극 활용. Kenney 팩은 음식·캐릭터 스프라이트도 다양해 커버 범위가 넓음
2. **직접 제작 (아이덴티티 에셋)**: 마스코트·음식 아이콘 24종 등 핵심 에셋은 Piskel(웹 기반 무료)·Aseprite로 제작. 16×16은 아이콘당 제작 부담이 작아 현실적. 후처리(팔레트 통일·투명화·정수배 스케일·스프라이트 시트 합치기)는 기존 Python/Pillow 파이프라인 재사용
3. **(보조) SVG 도트 시뮬레이션**: 단순 도형에 가까운 에셋(프레임·배지·버튼)은 이미지 없이 CSS/SVG 픽셀 테크닉으로 코드 생성 — Claude Code가 직접 만들 수 있어 제작 비용 0

> 원칙: 특정 게임의 에셋·캐릭터를 카피하지 않는다. "영감"은 스타일(타일 감성·팔레트·비례)까지만. 유료 생성 도구(Higgsfield 등)는 크레딧 이슈로 이 프로젝트에서 제외.

**연동 방식 (인증 불필요, 무료):**
- 시트를 "링크가 있는 모든 사용자 — 뷰어" 로 공개
- `https://docs.google.com/spreadsheets/d/{SHEET_ID}/gviz/tq?tqx=out:csv&sheet=restaurants` 로 CSV fetch
- Next.js API Route(`/api/restaurants`)에서 fetch + 파싱 + **10분 캐시** 후 프론트에 전달
- 시트 URL/ID는 서버 환경변수로만 보관 (외부 노출 방지)

### 4.2 카카오 카테고리 → 자체 카테고리 매핑 테이블

```
카카오 category_name (문자열 포함 매칭)     →  자체 카테고리
"음식점 > 한식 > 국밥", "탕", "설렁탕"       →  한식 / 국밥·탕
"음식점 > 한식 > 육류,고기"                 →  한식 / 고기구이
"음식점 > 일식 > 초밥,롤", "회"             →  일식 / 초밥·회
"음식점 > 양식 > 이탈리안", "피자"           →  양식 / 파스타·피자
... (전체 테이블은 개발 Phase 1에서 코드로 확정)
매칭 실패                                   →  기타 / fallback
```

- 매칭은 **뒤쪽(세부) 카테고리부터** 키워드 검사, 실패 시 대분류만이라도 매칭, 그것도 실패하면 "기타"
- 관리자DB 맛집과 카카오 결과가 **같은 가게**인 경우(이름+좌표 근접 매칭) → 관리자DB 정보로 덮어쓰기 (중복 제거 + 강조 표시)

### 5.3 비용 폭발 방지 장치 (설계에 내장)

1. **API Route 캐싱**: 카카오 검색 결과를 좌표 격자(약 100m) + 반경 단위로 5분 캐시. 동료 10명이 같은 사무실에서 돌려도 실제 카카오 호출은 몇 건 수준
2. **키 노출 차단**: 카카오 REST 키·기상청 키는 전부 Next.js API Route 뒤에. 프론트에 노출되는 건 카카오맵 JS 키뿐 → **카카오 콘솔에서 허용 도메인 등록 필수**
3. **사용량 알림**: 카카오 개발자 콘솔에서 쿼터 모니터링 습관화

## 6. 기술 병목 분석 & 통과 가이드 (7개)

### 병목 1 — 시그니처 메뉴·단체석 데이터 없음 ⭐ 최대 병목
- **문제**: 카카오 로컬 API는 상호명·카테고리·주소·좌표·전화번호만 반환. 메뉴/가격/**단체석·혼밥석 여부** 없음. 카카오맵 웹 크롤링은 약관 위반 + IP 차단 리스크
- **통과 가이드**:
  - 시그니처 메뉴·가격·단체석 정보는 **관리자DB(구글 시트)를 유일한 소스**로 삼는다
  - 결과 카드를 2등급으로 설계: `👑 관리자 추천` = 메뉴·가격·코멘트·단체석 풀 표시 / `일반 결과` = 카테고리·거리만 표시
  - **팀회식 모드는 관리자DB 주도**: group_seating=TRUE 등록 업장이 메인 후보. 카카오 결과는 회식형 카테고리(고기구이·횟집·중식당·이자카야)만 '단체석 미확인' 라벨로 보조 후보 처리 → 회식 장소로 등록 업장이 쌓일수록 모드 신뢰도 상승
  - **혼밥 모드는 카테고리 휴리스틱 주도**: 좌석 형태 데이터가 없어도 카테고리로 혼밥 적합도를 상당히 정확히 추정 가능 (국밥·라멘·덮밥은 혼밥 OK, 고기구이·요리 중식당은 다인 전제). 관리자DB solo_friendly는 휴리스틱을 뚫는 예외 등록용 (예: 혼밥 가능한 고깃집 — 1인 화로 있는 곳)
  - 이 구조가 오히려 "관리자 큐레이션 강조" 요구사항을 자연스럽게 충족
  - 운영 팁: 동료들이 "이 집도 넣어줘" 하면 시트에 한 줄 추가 → 즉시 반영 (재배포 불필요). 회식 다녀온 집은 group_seating·capacity를 그때그때 기록하는 습관이 팀회식 모드의 품질을 결정

### 병목 2 — 예산 필터의 데이터 공백
- **문제**: 가격 정보가 API에 없어 예산 필터가 카카오 일반 결과에 정확히 작동 불가
- **통과 가이드**:
  - 관리자DB 맛집: `price_tier` 필드로 정확 필터링
  - 카카오 일반 결과: **카테고리 기반 추정 테이블** 적용 (국밥·분식·쌀국수→가성비 / 고기구이·초밥→맛집 / 스테이크·오마카세→플렉스). 추정치임을 UI에 `~` 표기로 암시
  - "플렉스" 필터 선택 시엔 추정 오차가 크므로 관리자DB 우선 노출 가중치를 높인다

### 병목 3 — ~~Geolocation은 HTTPS 필수~~ → v1.9에서 위치인식 제거로 해소
- **결정 (v1.9)**: 사내용이라 위치인식을 아예 쓰지 않고 **시작점을 군인공제회관으로 고정**. Geolocation 권한 요청·HTTPS 의존·권한거부 fallback·지역경계 판정이 전부 불필요해져 병목 자체가 소멸.
- **시작점 좌표**: 군인공제회관(도곡동) — Phase 0에서 카카오 지오코딩으로 확정, 현재 근사값 lat 37.484, lng 127.034 (`src/lib/geo.ts` `COMPANY_COORDS`).
- **행정구역 필터는 유지**: 시작점은 고정이지만, 거리 모드별 후보의 행정구역 제한(도보/따릉이=서초·강남, 택시=+동작·송파)은 그대로 적용.

### 병목 4 — 날씨 자동연동 (기상청 API의 악명높은 구조)
- **문제**: 기상청 API는 위경도가 아닌 **자체 격자좌표(nx, ny)** 를 쓰고, base_time 규칙이 까다로움 (초단기실황은 매시 40분 이후 조회 가능 등). 특보는 별도 API
- **통과 가이드**:
  - 지역 한정 = 격자 고정값 하나로 해결 (**군인공제회관(양재역 인근) 기준 강남·서초 경계 격자, 대략 nx=61, ny=125~126** — 정확 값은 기상청 격자 엑셀에서 Phase 0에 확정). 지역 고정이라 위경도→격자 변환 로직 자체를 생략 가능 → 구현 단순화
  - `초단기실황(getUltraSrtNcst)` → 현재 강수 여부(PTY), 기온(T1H)
  - `기상특보조회(getWthrWrnList)` → 폭염·호우·한파 특보 (서울 지역코드 108/109 계열)
  - API Route에서 **30분 캐시** — 날씨는 자주 안 바뀜, 기상청 응답 지연 대비
  - 기상청 API 장애 시 fallback: 날씨 기능만 조용히 비활성 (앱 전체가 죽지 않게 try-catch 격리)
  - 악천후 판정 규칙: `폭염특보 OR 한파특보 OR PTY≠0(강수 중) OR 강수확률 60%↑` → **v1.14: 판정 결과는 안내·추천에만 쓴다** (하단 마스코트 말풍선·비 마스코트·날씨 기반 메뉴 추천). ~~거리 기본값 자동 전환~~ 폐지 — 사용자가 고른 필터를 앱이 말없이 바꾸지 않는다.

### 병목 5 — 거리 필터의 "실제 시간" 문제
- **문제**: 도보 10분·따릉이 거리를 정확히 하려면 길찾기 API(호출당 과금 가능)가 필요
- **통과 가이드 (v1.9)**:
  - 반경 컷은 **군인공제회관 기준 하버사인 직선거리 상한**: 도보 1.3km / 따릉이 2km / 택시 5km — 길찾기 API 불필요, 비용 0
  - 결과 카드 "도보 약 N분"은 표시용 근사만: 직선거리 × 도로보정 1.3 ÷ 67m/분 (반경 컷에는 보정 미적용, 순수 직선거리)
  - **관리자DB `access_mode` 오버라이드 (v1.11)**: 언덕·도로 사정으로 직선거리 판정이 실제와 어긋나는 곳은 시트 T열에 최소 이동수단(1=도보/2=따릉이/3=택시)을 지정. 값이 있으면 직선거리를 무시하고 `선택 등급 ≥ access_mode`로 노출, 비우면 기존 직선거리 반경 컷. 관리자DB 후보에만 적용(카카오 결과는 값이 없어 자동으로 직선거리 로직)
  - **이동수단 선택을 확률에도 반영 (v1.15)**: 반경 컷만으로는 택시(5km)가 도보권(1.3km)을 통째로 포함해, 모드를 바꿔도 결국 코앞 가게만 나왔다. 두 축을 곱해 보정한다.
    - `access_mode` 일치 → **×3** (`ACCESS_MODE_MATCH_BOOST`) — 관리자가 "여긴 택시로"라고 지정했으면 택시를 골랐을 때 잘 나오게
    - 거리 선호 → **×1~2** (`DISTANCE_PREF_BOOST`, 반경 기준 상대값): 도보=가까울수록↑ / 택시=멀수록↑ / 따릉이=중립
    - 실측 효과(관리자DB 기준): 0~500m 당첨확률 도보 48.7% ↔ 택시 16.6%, 1km 초과 도보 4.9% ↔ 택시 30.1%
    - **한계**: 시트가 반경 1500m로만 시드돼 **2km 초과 후보가 0곳**이다. 택시의 '멀수록↑'는 부스트할 대상이 사실상 없어, 지금은 `access_mode=3` 지정분(10곳)이 택시 모드의 실질적 차별점이다. 택시 모드를 제대로 살리려면 시트를 3~5km까지 다시 시드해야 한다.
  - 정밀 길찾기가 필요해지면 P2에서 카카오모빌리티 API 검토 (그때 비용 재점검)

### 병목 6 — 구글 시트 성능·안정성
- **문제**: 매 요청마다 시트를 읽으면 느리고(1~3초), 구글 측 rate limit 존재
- **통과 가이드**:
  - Next.js API Route에서 **메모리 캐시 10분** (Vercel 서버리스 특성상 콜드스타트 시 재fetch — 지인용 트래픽에선 문제없음)
  - CSV 파싱 시 빈 행·형식 오류 행은 건너뛰고 콘솔 경고 (시트 오타가 앱을 죽이지 않게)
  - `active=FALSE` 필터를 파싱 단계에서 적용
  - 좌표(H, I열)가 비어 있으면 카카오 지오코딩 API로 채우고 캐시에 보관

### 병목 7 — 카카오 카테고리 매핑의 불완전성
- **문제**: 카카오 카테고리 문자열이 일관되지 않음 (같은 국밥집이 "음식점 > 한식"까지만 달려 있기도 함)
- **통과 가이드**:
  - 키워드 포함 매칭을 **세부→대분류 순으로 폴백** 설계
  - 최종 실패는 "기타"로 — 제외 필터에서 "기타"도 선택 가능하게
  - 관리자DB 맛집은 시트에서 카테고리를 직접 지정하므로 이 병목의 영향 없음
  - 이름+좌표 근접(50m 이내) 매칭으로 카카오 결과와 관리자DB 중복 제거

### 9.1 Claude 토큰 사용량 최소화 전략 (v1.8)

**전제 확인 — 런타임 토큰은 0**: 이 앱은 배포 후 Claude API를 전혀 호출하지 않는다 (카카오·기상청·구글 시트만 사용). 토큰이 소모되는 유일한 구간은 **개발 단계의 Claude Code 사용량**이며, 아래 규칙으로 관리한다. "AI 추천 코멘트" 같은 Claude API 런타임 기능은 MVP에서 명시적으로 제외 — 추가하는 순간 사용자당 호출 과금이 생기므로 배포 후에도 도입 시 별도 비용 검토 필수.

**개발 단계 절약 규칙 (Claude Code)**

1. **서브에이전트는 필요할 때만** — §7.3의 4-에이전트 병렬 구조는 각 에이전트가 컨텍스트를 별도로 소비하므로 토큰이 배수로 든다. 실제 운용 원칙:
   - 기본은 **메인 세션 1개로 순차 진행** (A→B→C→D 순서). 지인용 규모 프로젝트는 이걸로 충분
   - 병렬 서브에이전트는 **명확히 독립적이고 큰 작업 2개가 동시에 필요할 때만** (예: 카카오 연동 + 기상청 연동 동시 개발)
   - 서브에이전트 결과는 요약본만 메인에 회수 (전체 코드 덤프 회수 금지)
2. **CLAUDE.md는 경량 요약본으로** — 이 기획안 전체(400줄+)를 CLAUDE.md에 넣으면 매 턴마다 컨텍스트로 소비된다. 대신:
   - CLAUDE.md에는 **핵심 스펙 요약 30~40줄**만 (모드 4종, API 3종, 시트 스키마, 카테고리 체계, 병목 회피 규칙)
   - 전체 기획안은 `docs/plan.md`로 두고 "상세 필요 시 해당 섹션만 읽어라"고 지시 — Claude Code가 필요한 부분만 부분 읽기
3. **Phase별 세션 분리** — Phase 1이 끝나면 새 세션으로 시작 (이전 대화 누적 컨텍스트 절감). 세션 간 인수인계는 `PROGRESS.md`에 완료 항목·다음 할 일 10줄 요약으로
4. **재작업 루프 차단이 최대 절약** — 토큰을 가장 많이 태우는 건 모호한 지시→재작업 반복. 이 기획안이 이미 그 방지책: 시트 스키마·API 스펙·모드별 로직이 확정돼 있어 "한 번에 맞게" 구현 가능. 개발 중 스펙 변경은 코드 먼저 고치지 말고 기획안 먼저 수정 후 반영
5. **결정적 작업은 스크립트로** — 스프라이트 시트 합치기, 팔레트 정리, 카테고리 매핑 테이블 생성 같은 반복 작업은 Claude에게 매번 시키지 말고 **한 번 스크립트로 만들어 재실행** (Pillow 파이프라인 방식). AI 재생성 반복은 토큰 낭비의 주범
6. **모델 선택** — 보일러플레이트·단순 CRUD·CSS 손질은 Sonnet으로 충분, 아키텍처 설계·병목 로직(카테고리 매핑, 모드별 후보 구성)만 상위 모델 사용
7. **테스트는 로컬 자동화로** — 필터 조합 테스트를 Claude에게 수동으로 반복시키지 말고 Phase 1에서 간단한 테스트 스크립트를 만들어 Claude는 실패 케이스만 보게 한다

## 11. 관리자 통계 대시보드 (v1.13)

`/stats` — 방문·룰렛·만족도 통계를 보는 관리자 전용 페이지. 페이지 내 비밀번호 입력창으로 진입.

### 11.1 왜 자체 수집인가
Vercel Web Analytics는 무료로 방문자 수를 주지만 **vercel.com 대시보드에서만** 보이고 조회 API가 없다.
룰렛 실행·당첨 가게·좋아요 같은 앱 고유 지표는 애초에 담기지 않는다. → 직접 쌓는다.
저장소는 **구글 시트 `stats` 탭** — 이미 동작하는 Apps Script 웹훅을 재사용해 새 인프라가 0이다.

### 11.2 ⚠️ Apps Script 선행 조건 (안 하면 관리자DB 오염)
기존 `doPost`는 허용 목록에 없는 탭이 오면 **에러 대신 restaurants로 조용히 폴백**한다.
이 상태로 `sheet:'stats'`를 보내면 방문할 때마다 restaurants 탭에 이벤트 행이 쌓인다.
→ `ALLOWED`에 `'stats'` 추가 + **폴백 제거(모르는 탭은 에러)** 후 **새 배포**가 선행되어야 한다.
코드 쪽 방어: `/api/track`은 응답의 `sheet`가 `stats`가 아니면 즉시 비활성화하고 더 쓰지 않는다.
`STATS_ENABLED` 환경변수로 이중 잠금 — 셀프테스트 통과 후에만 TRUE로 켠다.

### 11.3 stats 탭 스키마 (7열, A~G)
| 열 | 필드 | 값 |
|---|---|---|
| A | `ts` | ISO8601 (KST) |
| B | `event` | `visit` / `spin` / `like` / `map` / `reject` |
| C | `visitor` | 익명 ID (localStorage 난수. 개인정보 아님) |
| D | `mode` | `lunch-solo` / `lunch-group` / `dessert` / (visit은 빈칸) |
| E | `place` | 당첨 가게명 |
| F | `category_sub` | 당첨 하위 카테고리 |
| G | `detail` | `respin=1;price=보통;dist=walk;visited=1` 형태 (spin만) |

### 11.4 수집 이벤트
- `visit` — 첫 진입 1회 (세션당). 일별·누적 방문자 = visitor distinct
- `spin` — 룰렛 당첨 시. 모드·당첨 가게·카테고리·필터·재추첨 여부
- `like` — 결과 카드 **좋아요 버튼(신규 UI)**. 만족도 = like ÷ spin
- `map` — 결과 카드 '네이버지도에서 보기' 클릭. = map ÷ spin
- `reject` (v1.17) — 결과 카드 **'🎰 다시 돌리기'** 클릭 시, 그 순간 화면에 떠 있던(=버려진) 가게. `place`=버려진 가게. **기존 `respin` 플래그와 방향이 반대** — `respin=1` spin은 재추첨으로 *새로 뽑힌* 가게에 붙지만, `reject`는 *버려진* 가게를 가리킨다. 기피 신호로 수기 제외 판단에 사용. 시트 열 추가 없이 `event` 값만 추가(스키마 §11.3 불변).

수집은 클라이언트 → `/api/track`(fire-and-forget) → 서버가 웹훅 호출. 시크릿은 서버에만.
**개발 환경(NODE_ENV≠production)에서는 기록하지 않는다** — 로컬 테스트가 통계를 오염시키지 않도록.

### 11.5 대시보드 지표
- 요약: 누적/오늘 방문자, 총 룰렛, **좋아요율**, **지도 클릭율**, 재추첨률, **기피율**(reject ÷ spin)
- 일별 추이 (방문자·룰렛)
- 모드별 룰렛 분포
- 당첨 가게 TOP 10 / 카테고리 분포
- **기피 식당 TOP 10** — 버림 횟수 + 노출(당첨) 대비 기피율. 노출 적어 우연히 버려진 곳과 자주 떠도 자주 버려지는 지뢰를 구분
- 필터 사용 분포 (예산·거리·인증우선)

### 11.6 인증
`STATS_KEY` 환경변수. `/stats` 페이지에서 비밀번호 입력 → `/api/stats`가 검증(길이 무관 상수시간 비교).
통과 시 sessionStorage에 보관해 새로고침 시 재입력 불필요. 페이지 자체는 공개지만 **데이터는 API가 지킨다**.

### 11.7 한계 (알고 쓰는 것)
- 시트는 이벤트 로그에 최적이 아니다. 하루 수백 건 수준까지가 안전선. 그 이상이면 KV로 이전.
- Apps Script 동시 append 경합 가능성 (사내 트래픽에선 무시 가능).
- JS를 실행하는 클라이언트만 기록 → 크롤러는 자연히 제외되지만, 정확한 UV는 아님.

## 12. 예비 시트(candidates) 파이프라인 (v1.16)

### 12.1 문제 — 카카오 45건 한계
카카오 로컬 검색은 **한 질의당 최대 45건**(3페이지×15)만 준다. `meta.total_count`는 진짜 총계를
알려주지만 그 이상은 못 가져온다. 그래서 `/api/sync`가 매일 돌아도 신규 매장을 사실상 못 찾았다.

| 반경 | `total_count` | 기존 회수 | 격자 스캔 회수 |
|---|---|---|---|
| 200m | 37 | 37 | 37 |
| 1,300m | **844** | 45 (5%) | **845** |
| 1,500m | 1,365 | 45 (3%) | — |
| 2,000m | 3,479 | 45 (1%) | **2,617** (4개구 필터 후) |

### 12.2 해법 1 — 적응형 격자 스캔 (`kakaoScan.ts`)
반경 200m면 45건 미만이라 전부 회수된다. 영역을 **사각 쿼드트리**로 쪼개 각 셀이 45건 미만이 되게 한다.
- 셀(정사각형)을 덮는 원의 반경 = 반변길이 × √2
- `total_count > 45`인 셀만 4등분 재귀 → 밀집 지역만 깊게, 한산한 곳은 1회로 끝
- 안전핀: `MIN_HALF_SIDE_M`(60m) 이하로는 안 쪼갬 · `maxCalls`(기본 400) 상한 · 동시 6
- 실측: 반경 2km에서 **466회 호출 · 9.2초 · 포화 셀 0개**. Vercel Hobby 함수 상한 300초라 여유.

`kakao.ts`의 `searchNearby`(45건·5분 캐시)는 **룰렛 실시간 검색용으로 그대로 둔다** — 룰렛은
반경 내 일부만 있어도 되고, 격자 스캔은 느리고 호출이 많아 매 요청에 쓸 수 없다.

### 12.3 해법 2 — 오염·비용 격리
2,000곳을 `restaurants`에 그대로 넣으면 손으로 큐레이션한 236행이 미검증 행에 묻힌다.
→ **예비 시트 `candidates` 신설**. 룰렛은 이 시트를 보지 않는다.

```
카카오 격자 스캔(무료·크론) → candidates → 구글 검증(수동·배치) → pass만 → restaurants(룰렛)
```

### 12.4 candidates 스키마 (24열)
**A~T는 restaurants 20열과 완전히 동일** (승격 = 앞 20열 복사) + `google_rating`(U) /
`google_reviews`(V) / `verdict`(W: 빈칸·pass·fail·miss·promoted) / `checked_at`(X).
`checked_at`은 `stats.ts`의 `toKstStamp()` 재사용 — ISO로 쓰면 시트가 날짜 셀로 바꿔 깨진다(§11.3).

### 12.5 구글 교차검증 — 비용 구조 (2026-07 확인)
카카오 응답엔 평점·리뷰수 필드가 **아예 없다**(확인함). 구글은 있지만
`rating`/`userRatingCount`가 **Enterprise 등급**이라:

| SKU | 단가 | 무료/월 |
|---|---|---|
| Text Search **Enterprise** (평점 O) | **$35/1,000** | **1,000회** |
| Text Search Pro (평점 X) | $32/1,000 | 5,000회 |

**절대 규칙 3가지** — 안 지키면 월 $2,000이 나온다:
1. **수동 실행 전용** (`scripts/verify-candidates.mts`), 배치 상한 필수
2. **키는 `.env.local`에만.** Vercel엔 넣지 않는다 → 서버가 호출할 경로 자체를 없앤다
3. **탈락분도 시트에 기록**(`verdict=fail`) → 재조회 안 함. 이게 없으면 매일 2,000건 재조회

### 12.6 조용한 폴백 2종 (이 프로젝트를 두 번 문 함정)
- **Apps Script**: 모르는 탭 → 에러 대신 restaurants에 씀 → `ALLOWED` + 폴백 제거 + 응답 `sheet` 검증
- **구글 시트 gviz**: 없는 탭 요청 → **404가 아니라 200 + 첫 번째 탭 내용** → `candidatesSheet.ts`가
  헤더 W열이 `verdict`인지 확인해 막는다. 안 막으면 restaurants 235행을 '미검증 후보'로 착각해
  구글 조회비를 태우게 된다.

### 12.7 운영 절차
1. 크론(매일 정오) → `/api/sync` → 격자 스캔 → candidates에 신규 append
2. 수동: `npx tsx scripts/verify-candidates.mts 100 --max-dist 800`
3. pass → restaurants 자동 승격 (승격 직전 중복 재확인 → 재실행 안전)
4. 승격 후 손으로: `signature_menu` / `solo_friendly` / `access_mode` / `visited`·`rating`

상세는 `docs/candidates-setup.md`.
